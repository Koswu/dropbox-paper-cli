# Implementation Plan: Replace Dropbox SDK with Direct HTTP API + httpx

**Branch**: `002-httpx-api-migration` | **Date**: 2025-07-15 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/002-httpx-api-migration/spec.md`

## Summary

Replace the `dropbox` Python SDK with direct Dropbox HTTP API calls via `httpx` AsyncClient. All service methods become `async def`, with `asyncio.run()` at each Typer CLI entry point bridging syncв†’async. The sync orchestrator migrates from `ThreadPoolExecutor` to `asyncio.gather()` + `Semaphore(20)`. Token refresh uses `asyncio.Lock` with double-check pattern. Two timeout profiles cover metadata (5s/5s) and content (5s/30s) operations. DEBUG-level HTTP logging is gated behind `--verbose`.

## Technical Context

**Language/Version**: Python 3.12+ (required by constitution)
**Primary Dependencies**: httpx (replaces dropbox SDK), typer (CLI framework, unchanged), textual (TUI, unchanged)
**Storage**: SQLite via built-in `sqlite3` (metadata cache, unchanged)
**Testing**: pytest + pytest-asyncio (existing; all service tests migrate to async)
**Target Platform**: Linux, macOS (primary); Windows best-effort
**Project Type**: CLI tool
**Performance Goals**: Sync 500+ items в‰Ґ30% faster than current SDK+ThreadPoolExecutor approach
**Constraints**: Metadata API calls <5s, content operations <30s; max 20 concurrent requests (configurable)
**Scale/Scope**: Single-user CLI tool, ~2k LOC in services layer being rewritten
